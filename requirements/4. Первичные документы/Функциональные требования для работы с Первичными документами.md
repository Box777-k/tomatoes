| Номер | Краткое название функции приложения (web app) | Описание в 3–4 предложения                                                                                                                                                                                                                                                                                                                                                                |
| ----- | --------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 4.1   | Создание первичного документа                 | Пользователь создаёт первичный документ выбранного типа (поступление, реализация, акт услуг, перемещение, списание и т.п.). Документ всегда создаётся в контексте организации и даты. Документ является основанием для движений по складу, деньгам и бухгалтерии. Должны поддерживаться статусы документа.                                                                                |
| 4.2   | Строки документа                              | Пользователь добавляет строки документа: номенклатура, количество, цена и сумма. Система должна автоматически считать итоги и проверять единицы измерения. Для документов движения запасов обязательны склад/откуда-куда и при необходимости МОЛ. Изменение строк должно фиксироваться в истории.                                                                                         |
| 4.3   | Проведение документа                          | Пользователь проводит документ, после чего система создаёт связанные движения (склад/деньги/проводки) по правилам типа документа. Проведённый документ нельзя редактировать напрямую, только через отмену проведения или корректировку. При проведении выполняются проверки остатков, лимитов и обязательных реквизитов. Результат проведения должен быть прозрачно показан пользователю. |
| 4.4   | Отмена проведения                             | Пользователь может отменить проведение документа при наличии прав. Система должна корректно отменять все связанные движения и возвращать документ в черновик. Отмена проведения должна оставлять след в журнале аудита. Для закрытых периодов отмена должна быть запрещена или требовать специальной роли.                                                                                |
| 4.5   | Связь документа с платежами                   | Пользователь связывает документ с одним или несколькими платежами (полная или частичная оплата). Система показывает статус оплаты и остаток к оплате. Связь используется для контроля задолженности и отчётности. Должна поддерживаться привязка к гранту через платежи или напрямую.                                                                                                     |
| 4.6   | Связь документа со складскими движениями      | Система должна показывать, какие складские движения созданы документом и их статус. Пользователь должен видеть, какой склад/МОЛ затронут и какой остаток изменился. Для документов перемещения должны быть подтверждения «отпустил/принял» при необходимости. Должна быть возможность перейти из движения к исходному документу.                                                          |
| 4.7   | Приложения и сканы                            | Пользователь прикрепляет файлы к документу (сканы, фото, PDF, счета, акты). Файлы используются для подтверждения расходов и отчётности, включая гранты. Должны поддерживаться контроль доступа и хранение истории версий. Система должна позволять быстро открыть файл из карточки документа.                                                                                             |
| 4.8   | Нумерация документов                          | Система присваивает документам номера по правилам организации и типа документа. Должны поддерживаться отдельные последовательности для разных организаций. Номер фиксируется при проведении или по настройке при создании. Запрещены дубли номеров в рамках одной серии.                                                                                                                  |
| 4.9   | Журнал документов                             | Пользователь видит список документов с фильтрами по типу, организации, складу, контрагенту, периоду и статусу. Журнал должен быстро открывать карточку документа и показывать ключевые итоги. Должны быть быстрые действия: провести, отменить проведение, распечатать/экспортировать. Должна быть возможность поиска по номеру и тексту.                                                 |
| 4.10  | Шаблоны печатных форм                         | Система формирует печатные формы для выбранных типов документов (накладная, акт, счёт и т.п.). Шаблоны должны быть настраиваемыми и привязанными к организации. Печатная форма должна соответствовать данным документа и фиксировать дату формирования. Должна быть возможность выгрузки в файл.                                                                                          |
